# AGENTS.md

## Cursor Cloud specific instructions

FinalStrike is a single Python package (`finalstrike`) exposing a Typer CLI. `fixtures/sample-app/` is the integration test target repo (not a separate product). Python 3.12 is used.

### Environment
- Dependencies are installed into a project virtualenv at `.venv` (gitignored). Activate with `source .venv/bin/activate` or prefix commands with `.venv/bin/` (e.g. `.venv/bin/pytest`, `.venv/bin/finalstrike`). The update script keeps `.venv` in sync.
- The test suite needs a gitignored local secrets vault at `fixtures/sample-app/.finalstrike/secrets.env` containing fake values `OPENAI_API_KEY=fixture-test-key-not-real` and `SLACK_BOT_TOKEN=fixture-slack-token`. Without it, 6 tests in `tests/test_p1_context.py` fail. The update script recreates it if missing.

### Lint / test / build / run
- Tests: `pytest -q` (config in `pyproject.toml` `testpaths = ["tests"]`). Run with the venv **activated** (`source .venv/bin/activate`) so `.venv/bin` is on `PATH`. The orchestrator's terminal-layer tests spawn a bare `pytest` subprocess, so invoking the suite as `.venv/bin/pytest` (without activation) leaves `pytest` off `PATH` and makes 3 tests in `tests/test_p3_runners.py` fail with `pytest: not found`.
- Lint: no linter is configured (dev deps are pytest only); there is nothing to run.
- Build/run the app: it is a CLI, not a server. Core commands: `finalstrike --version`, `finalstrike validate-config --repo fixtures/sample-app`, and `finalstrike plan --repo fixtures/sample-app --acceptance fixtures/sample-app/acceptance.md --dry-run`. See `README.md` Quick start.
- Regenerate JSON schemas: `python -m finalstrike.config.export_schemas` (writes to `schemas/`; output is committed and currently up to date).
- The `fixtures/sample-app` server (integration target) runs via `python -m sample_app.server 8080` from inside `fixtures/sample-app/` and serves `GET /health` -> `200 ok`.

### Live LLM testing (external API)
- LLM config is read from `fixtures/sample-app/finalstrike.yaml` under `llm:` (`provider: openai_compat`, `base_url`, `model`). The transport is a single OpenAI-compatible client, so any gateway works by setting `base_url`/`model` (OpenAI's public `/v1` API, an OpenRouter URL, a LiteLLM proxy, etc.). The committed fixture points at a local Ollama example (`http://localhost:11434/v1`, `llama3`); no local LLM server is installed on this VM.
- Live external-API testing requires a real `OPENAI_API_KEY` Cloud Agent secret (optionally `FINALSTRIKE_LLM_BASE_URL`/`FINALSTRIKE_LLM_MODEL` for a gateway). Outbound HTTPS egress works (the OpenAI `/v1/models` endpoint returns 401 without a key, and an OpenRouter `/models` endpoint returns 200) and the `openai` SDK is in `.venv`. A real authenticated completion has been verified working through both the raw SDK and FinalStrike's own config loader.
- GOTCHA (key precedence) â€” the vault OVERRIDES the process env, not the other way around: `apply_to_environ` does `env.update(secrets)` and `load_repo_context` does `os.environ.update(secrets)`. So running live through `fixtures/sample-app` resolves the vault's FAKE `OPENAI_API_KEY` and 401s, even when a real `OPENAI_API_KEY` is exported. Do NOT fix this by editing the committed fixture vault â€” `tests/test_p1_context.py::test_load_secrets_from_sample_app` hard-asserts `OPENAI_API_KEY == "fixture-test-key-not-real"`, so changing it breaks `pytest -q`. Instead, run live against a SEPARATE `--repo` (e.g. a throwaway dir under `/tmp`) whose own `finalstrike.yaml` (`llm:` base_url/model) and `.finalstrike/secrets.env` (real key) you create at runtime; that leaves the committed fixture and tests untouched.
- GOTCHA (request param) â€” the configured OpenAI model rejects `max_tokens` with HTTP 400 (`Unsupported parameter ... Use 'max_completion_tokens'`). The P5 planner/provider adapter must send `max_completion_tokens`, and reasoning-style models may need a generous budget (reasoning tokens count toward it) plus default temperature.
- STATUS â€” the live LLM planner (Phase 5) is NOT implemented. `finalstrike/planner` and `finalstrike/providers/openai_compat.py` are empty stubs, so `finalstrike plan --no-dry-run` only prints the merged context with a "LLM planner is not implemented yet" note and makes no API call. There is no `finalstrike doctor` command, no `generate_verification_plan`, no `fixtures/sample-app/acceptance-smoke.md`, no `tests/test_p5_*`, and no `docs/PHASE_GAPS.md`/`tests/llm_recordings/`. Until P5 lands there is no first-class CLI path that calls the external API (the verified live calls above were made directly against the resolved config).
